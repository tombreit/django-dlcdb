{% load i18n %} {% comment %} Reusable live-search picker (person, device, room, …), driven by theme/js/picker.js. Shared across frontend apps; the caller supplies the search endpoint and the templates that render the options. Parameters (via {% include ... with %}): picker_id unique slug; all element ids derive from it field_id id of the hidden form field this picker writes into search_url HTMX endpoint returning the result options label visible field label placeholder search input placeholder required show the red required marker locked disable selecting/changing (read-only) fill_date_target (optional) id of a date input to prefill from the option's data-contract-end sync_toggle_target (optional) id of a checkbox that is auto-checked when fill_date_target gets prefilled (marks the date as contract-derived); see picker.js errors (optional) bound-field errors to render search_name (optional) name of the search input (default "search"); use a distinct value when two pickers share one
so HTMX does not mix their query terms hx_vals (optional) JSON string posted with every live search via hx-vals (e.g. '{"source": "lend"}'); lets one shared search endpoint know which data source to use has_selected truthy when a card is pre-selected (collapses the search) selected_item_template (optional) template path rendering the pre-selected card (the object it expects must be passed in too, e.g. person=…) multiple (optional) allow selecting several options; each becomes a card carrying its own hidden input (see picker.js). The search stays open and there is no single hidden field. field_name (multiple only) name for the per-card hidden inputs selected_items (multiple only) iterable of pre-selected objects to render back after a validation error. Each is passed to selected_item_template as ``device`` (the device picker is the only multi-select consumer today). An element with id "-count" placed anywhere on the page (e.g. in a section header) is kept in sync by picker.js with the number of selected cards. {% endcomment %}
